# รายงานสรุปและเปรียบเทียบผลการทดลองกลุ่ม Sequential 2-Agent (Research Summary v2 - Exp 3)

รายงานฉบับนี้ทำการวิเคราะห์สรุปผลการทดลองในกลุ่ม **Sequential 2-Agent Pipeline (Exp 3, 3E, 3F)** รอบรันใหม่เปรียบเทียบกับ:
1. Baseline ดั้งเดิมใน **reportV1** เพื่อประเมินผลลัพธ์หลังการปรับปรุงระบบคำสั่ง (Prompt Optimization) ในระบบ sequential
2. เปรียบเทียบอย่างละเอียดกับกลุ่มสถาปัตยกรรม **Flat Classification (Exp 1, 1E, 1F)** และ **Two-Layer Joint Classification (Exp 2, 2E, 2F)** เพื่อประเมินการออกแบบระบบที่ดีที่สุดในการคัดกรองภัยพิบัติ

## 3. การวิเคราะห์เชิงลึก: ทำไมระบบ Sequential (2-Agent) จึงแพ้ในเชิงโครงสร้าง?

จากการวิเคราะห์สถิติสรุปผล เราพบสาเหตุหลัก 2 ประการที่ทำให้สถาปัตยกรรมแบบ **Sequential (Exp 3)** ขาดประสิทธิภาพเมื่อเทียบกับรูปแบบอื่น ๆ:

### 1. ❌ ปัญหาการสะสมความผิดพลาด (Error Propagation Bottleneck)
* ระบบ Sequential อาศัยเอเจนต์ตัวแรก (Agent 1) ทำหน้าที่ตัดสินใจตัดข้อมูลแบบ "กรองผ่านหรือไม่ผ่าน" (Binary Filter: informativeness)
* หาก Agent 1 คาดการณ์ผิดพลาด โดยประเมินว่าข้อความภัยพิบัติที่สำคัญเป็น "ไม่เกี่ยวข้อง" (False Negative) ข้อความนั้นจะถูกตัดออกจากระบบทันที ส่งผลให้เอเจนต์ตัวที่สอง (Agent 2) ไม่มีโอกาสจำแนกประเภท (Category) ได้ถูกต้อง
* ส่งผลให้ค่า **Informativeness F1 score** ตกลงไปอยู่ที่ช่วง `~0.81 - 0.84` สำหรับ 3E (ลดลงอย่างชัดเจนจากแบบ Joint และ Flat ที่ทำได้ `~0.88 - 0.90`) และทำให้ค่า **Category F1 score** ร่วงลงตามไปด้วยเนื่องจากโอกาสกู้คืนความผิดพลาดใน Agent 2 เป็นศูนย์

### 2. 📉 ความสิ้นเปลืองโทเค็นอย่างทวีคูณ (Token Inefficiency)
* ระบบ Sequential 2-Agent สิ้นเปลืองโทเค็นมากที่สุดในทุกการรัน 
* ในรอบ Few-Shot (Exp 3F) การเรียกประมวลผล 2 เอเจนต์แยกส่ง Request API (Agent 1 กรอง และ Agent 2 คัดแยกหมวดสำหรับข้อความที่เกี่ยวข้อง) ทำให้เกิดการส่ง System Prompt, คำอธิบายหมวดหมู่ และตัวอย่าง 16 เคสซ้ำซ้อนสองรอบ
* ตัวอย่างใน **Gemma-4** (Few-shot): แบบ Flat (1F) ใช้โทเค็นรวมเพียง **432,898** โทเค็น ขณะที่ Sequential (3F) พุ่งสูงถึง **746,161** โทเค็น (**เพิ่มขึ้นกว่า +72%**)
* ข้อจำกัดนี้พบได้ในทุกโมเดล (Deepseek-v4-flash ใช้เพิ่มขึ้น **+70%**, Typhoon-v2.5 ใช้เพิ่มขึ้น **+60%**)

## 4. ข้อเสนอแนะเชิงกลยุทธ์เพื่อนำไปใช้งานจริง (Strategic Recommendation)

1. **หลีกเลี่ยงสถาปัตยกรรมแบบ Sequential 2-Agent สำหรับงานคัดแยกข้อความ**:
   * แม้ว่าไอเดียเรื่องการแยก Agent ตรวจสอบทีละขั้นจะดูเรียบง่าย แต่ในทางปฏิบัติให้ผลลัพธ์ที่แย่กว่าในทุกมิติ ทั้งคะแนนความถูกต้องที่ลดลง (เนื่องจาก Error Propagation) และภาระค่าใช้จ่าย API ที่พุ่งสูงขึ้น 60% - 70%

2. **ใช้ Two-Layer Joint (Exp 2E) สำหรับการเจาะลึกหมวดหมู่ (สูงสุดทางประสิทธิภาพ)**:
   * หากเน้นความถูกต้องในแยกหมวดความช่วยเหลือภัยพิบัติสูงสุด แนะนำให้เลือก **`typhoon-v2.5` ร่วมกับสถาปัตยกรรม Two-Layer Joint (Exp 2E) @ Temp 0.2** (ได้ Category F1 สูงสุดถึง **0.6493**)

3. **ใช้ Flat Classification (Exp 1E) เพื่อความประหยัดและความรวดเร็ว (สูงสุดทางความคุ้มค่า)**:
   * การประยุกต์ใช้ **Flat Classification (Exp 1E) ร่วมกับโมเดล `typhoon-v2.5` หรือ `gemma-4`** เป็นตัวเลือกที่ดีที่สุดสำหรับการนำไปใช้ในระบบจำลองจริง เนื่องจากให้ค่าความแม่นยำ F1 รวมสูงมากใกล้เคียงกัน แต่ประหยัดการใช้โทเค็นและทรัพยากรไปได้ครึ่งหนึ่งเมื่อเทียบกับระบบ Sequential
