# แผนกลยุทธ์การปรับปรุงประสิทธิภายการคัดแยก (Optimization Plan to Reach 0.75 - 0.85 F1)

จากผลลัพธ์ของ **V2-1 (Hierarchy Prompt)** คะแนนเฉลี่ยของ Gemma-4 และ DeepSeek-v4-flash ขึ้นมาแตะช่วง **`0.62 - 0.64`** ซึ่งถือว่าอยู่ในเกณฑ์ที่ดีสำหรับการทำ Zero-shot / Simple Few-shot บนทวีตที่มีความกำกวมสูงอย่าง CrisisMMD

อย่างไรก็ดี การจะไต่ขึ้นไปถึงเกณฑ์ **`0.75 - 0.85`** (ซึ่งเป็นเกณฑ์ที่เทียบเท่าระดับความถูกต้องของมนุษย์ หรือ Human Agreement Rate ของชุดข้อมูลนี้) จำเป็นต้องเปลี่ยนผ่านจากคำสั่งแบบระบุพฤติกรรมธรรมดา ไปสู่ระบบประมวลผลเชิงตรรกะและการเรียนรู้เชิงบริบทขั้นสูง ดังนี้:

---

## 1. 🧠 กลยุทธ์ที่ 1: Chain-of-Thought (CoT) + Priority Flow (ตรรกะคิดทีละขั้นตามลำดับขั้น)
ปัจจุบันโมเดลถูกบังคับให้ทำนายคลาสทันทีผ่าน API (Flat หรือ Tool calling) ทำให้ขาดขั้นตอนคิดคำนวณน้ำหนัก

### แนวทางปฏิบัติ:
ให้โมเดลสร้าง "กระดาษทดความคิด" (Reasoning thoughts) ในการตอบคำถามทีละขั้นก่อนส่งผลลัพธ์สุดท้ายกลับมา โดยกำหนด Flowchart ใน Prompt ดังนี้:
1.  **Extract Evidence**: ระบุข้อความเชิงประจักษ์ในทวีต (เช่น "สะพานพัง", "รถยนต์จมน้ำ", "ตาย 2 คน")
2.  **Casualty Check**: ตรวจสอบว่ามีผู้เสียชีวิตหรือบาดเจ็บหรือไม่? (ถ้ามี -> ปักหมุดที่ `injured_or_dead_people` ทันทีและหยุดวิเคราะห์)
3.  **Displacement Check**: ตรวจสอบว่ามีการอพยพ ไร้ที่อยู่อาศัย หรือสูญหายหรือไม่? (ถ้ามี -> ปักหมุดที่คลาสที่เกี่ยวข้อง)
4.  **Structural vs. Vehicle Check**: เปรียบเทียบความเสียหายระหว่างโครงสร้างพื้นฐานกับยานพาหนะ
5.  **Output JSON**: สรุปคลาสสุดท้ายจากลำดับขั้นความสำคัญ

> 💡 **ทำไมถึงได้ผล?**: โมเดลขนาดย่อมไปจนถึงใหญ่ (Gemma-4 26B, DeepSeek-v4-flash) มีสติปัญญาเชิงเหตุผลสูงมาก การปลดล็อก CoT จะช่วยเพิ่มความแม่นยำ F1 ขึ้นอีกเฉลี่ย **+5% ถึง +8%**

---

## 2. 🔍 กลยุทธ์ที่ 2: Dynamic Few-Shot (Semantic RAG)
ปัจจุบันเราใส่ตัวอย่าง Few-shot แบบตายตัว (Static 16 เคส) หากทวีตที่ทดสอบเป็นประเด็นที่ต่างไป ตัวอย่างเหล่านั้นจะช่วยได้ไม่เต็มที่

### แนวทางปฏิบัติ:
สร้างตัวเลือกตัวอย่างภัยพิบัติ (Exemplar Pool) ขนาด 150-200 ข้อความ และใช้ **Semantic Similarity Search** (เช่น BM25 หรือ Sentence Transformers ขนาดเล็ก) ในการดึงตัวอย่างที่ใกล้เคียงที่สุด 3-5 เคสขึ้นมาหยอดใส่ Prompt แบบไดนามิกก่อนส่งให้ LLM:
*   *ตัวอย่าง:* หากทวีตทดสอบพูดถึง "เสาไฟล้มและไฟดับ" -> ระบบจะดึงทวีตตัวอย่างเกี่ยวกับการเสียหายของโครงสร้างสายไฟและ blackout มาเป็นตัวอย่างสอนโมเดลโดยเฉพาะ
*   *ผลลัพธ์:* โมเดลจะเห็นขอบเขตตัวอย่างการจำแนกประเภท (Decision Boundary) ที่ตรงกับประเด็นในทวีตจริงที่สุด

> 💡 **ทำไมถึงได้ผล?**: วิธีนี้ช่วยแก้ไขความกำกวมของคำศัพท์เฉพาะถิ่น ช่วยดัน F1-score พ้นเกณฑ์เพดานจำกัดทั่วไปขึ้นไปอีก **+6% ถึง +10%**

---

## ⚖️ กลยุทธ์ที่ 3: Error-Driven Hard Negatives (การจูน Prompt จี้จุดที่โมเดลสับสนบ่อย)
จากการวิเคราะห์ Confusion Matrix ของ V2 และ V2-1 พบว่าโมเดลยังมีจุดสับสนร่วมกันอย่างมีนัยสำคัญ:
1.  **Infrastructure vs. Vehicle Damage**: สะพานพังทับรถยนต์ (โมเดลตอบ Vehicle แต่ Ground truth เป็น Infrastructure)
2.  **Rescue/Volunteering vs. Affected Individuals**: "การแจกอาหารที่ศูนย์อพยพ" (โมเดลตอบ Affected แต่ Ground truth เป็น Rescue/Volunteering)

### แนวทางปฏิบัติ:
เพิ่มหัวข้อ **"⚠️ Borderline Resolution Cases"** ใน Prompt โดยใส่ตัวอย่างเคสที่สับสนบ่อยเหล่านี้ลงไปในคำแนะนำโดยตรง เพื่อเน้นย้ำจุดแบ่งเกณฑ์การให้คะแนนอย่างเจาะจง

---

## 🔀 กลยุทธ์ที่ 4: Sequential Self-Consistency Voting (การโหวตความสอดคล้อง)
เนื่องจากระบบ Sequential 2-Agent (3E) ปรับปรุงกติกา Bias Rule แล้วทำ F1 ของ Gemma พุ่งสูงสุดถึง `0.6430` เราสามารถนำกระบวนการ **Self-Consistency (Ensemble)** เข้ามาเสริมได้

### แนวทางปฏิบัติ:
1.  รัน Agent 1 ด้วยอุณหภูมิปกติเพื่อแยก Informativeness
2.  ส่งต่อทวีตให้ Agent 2 จำแนกประเภทหมวดหมู่ย่อย โดยกำหนด **Temperature = 0.3** และสั่งรันพร้อมกัน 3 ครั้ง (หรือ 3 Threads)
3.  ใช้ตรรกะ **Majority Vote** (โหวตเสียงส่วนใหญ่) ในการตัดสินหมวดหมู่ขั้นสุดท้าย เพื่อล้างสัญญาณรบกวน (Noise) หรือความแปรปรวนของการสุ่มตอบของโมเดล

---

## 🏋️ กลยุทธ์ที่ 5: Parameter-Efficient Fine-Tuning (PEFT/LoRA)
หากการทำ Prompt Engineering ทั้งหมดถึงทางตัน (มักติดเพดานที่ประมาณ `0.70` เนื่องจากสไตล์การป้ายกำกับของคนคัดเลือกข้อมูล CrisisMMD ที่บางเคสมีความไม่สม่ำเสมอ หรือ Subjective สูง):

### แนวทางปฏิบัติ:
*   นำชุดข้อมูล TSV ทั้งหมดในโฟลเดอร์ `dataset/crisis-mmd/` (ซึ่งมีหลายพันรายการ) มารวมกันและแบ่งทำ Train/Validation
*   ใช้เทคนิค **LoRA (Low-Rank Adaptation)** เพื่อเทรนโมเดล **Gemma-4** หรือ **DeepSeek-v4-flash-base** 
*   การ Fine-tuning จะช่วยให้โมเดลจำสไตล์การให้หมวดหมู่ย่อยและบริบทแวดล้อมเฉพาะตัวของ CrisisMMD ได้ดีที่สุด ซึ่งเป็นวิธีเดียวที่การันตีผลลัพธ์ **F1-score ในช่วง `0.78 - 0.85`**
